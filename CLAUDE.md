# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

`github.com/ranjbar-dev/hd-wallet` is a published, security-focused, **Trust Wallet–compatible** HD wallet library (single Go package `hdwallet`, plus a demo CLI in `cmd/hdwallet`). It derives addresses and signs digests/messages for **134 networks**; the registered coins span 7 elliptic curves (secp256k1, ed25519, nist256p1, ed25519-blake2b, curve25519, starkex/StarkNet, ed25519-extended/Cardano) and the package implements 8 curve schemes total (sr25519 is implemented but not yet wired to a registered chain). On top of raw signing it provides **EVM tooling** (RLP, ABI, EIP-191, EIP-712), **protobuf transaction signing** for many families (`SignTransaction`; EVM/Tron/XRP/Cosmos/Solana/Bitcoin-UTXO/Stellar/Algorand/Aptos, plus Bitcoin P2SH/P2WSH multisig — no broadcast), **secure private-key import/export**, and **address validation/parsing** (`AnyAddress`-style). **Correctness is fund-critical**: a wrong address or signature means permanently lost funds, so nothing in the derivation/encoder/signing paths ships without passing an authoritative test vector.

gofmt -l .                                 # must be empty
go vet ./...
golangci-lint run ./...                    # v2 config in .golangci.yml; CI pins v2.12
gosec -quiet ./...                         # security scan
govulncheck ./...                          # use Go ≥1.24.4 locally to avoid a stdlib-only false positive

There is no Makefile; run the Go toolchain directly. Tests live in the same package (`package hdwallet`) so they can exercise unexported helpers; examples/benchmarks are `package hdwallet_test`.

## Architecture

The whole library is one flat package built around three concerns. Reading these in order explains the system:

1. **Secret isolation (`secret.go`, `hdwallet.go`).** The mnemonic, derived seed, and (for mnemonic wallets) the BIP-39 entropy live only in **memguard enclaves** (encrypted at rest in RAM, page-locked against swap, auto-wiped) — never as a Go `string` or long-lived `[]byte`, and never in an exported field. `secret.withSeed(fn)` opens the seed enclave exactly once per operation and destroys the decrypted buffer on return; `secret.withEntropy(fn)` does the same for the Cardano entropy enclave (returning `ErrNoEntropy` for key-only/WIF/xpub wallets). All three enclaves are cleared in `destroy()`. `HDWallet` guards everything with a `sync.RWMutex` and a `w.secret == nil → ErrDestroyed` check; `Destroy()` wipes secrets. **Invariant: private keys are derived, used, and wiped inside the package — they are NEVER returned to callers.** There is deliberately no `PrivateKey()` getter; adding one would defeat the entire design.

2. **Curve-agnostic registry (`registry.go`).** `Symbol` is a typed string enum with an exported constant per network (`BTC`, `ETH`, `SOL`, …). `Coin{Name, Symbol, Curve, Path, Encode}` is the registry row; `coins` is a `map[Symbol]Coin`. `Curve` is one of `Secp256k1`, `Ed25519`, `Nist256p1`, `Ed25519Blake2bNano`, `Curve25519`, `Starkex`, `Ed25519ExtendedCardano` (registered curves; `Sr25519` is implemented but not registered). **Adding a network is a single registry row** plus an encoder — the public API (`Address`, `Sign`, `PublicKey`, `AllAddresses`, `SupportedCoins`) is data-driven off this map.

3. **Derivation + signing layers** — designed so future per-chain transaction builders ("Option B") reuse the bottom layers without changes:
   - **Layer 1 — `derive.go`:** `withPrivateKey(seed, coin, fn)` is the *single* place a private key is materialized; it derives the leaf key per curve, hands the raw 32-byte key to `fn`, and **wipes it on return**. `derivePublicKey` is implemented on top of it. secp256k1 uses BIP-32 via `btcd/hdkeychain`; ed25519 and nist256p1 use **SLIP-0010** (`slip10.go`, `deriveEd25519`/`deriveNist256p1`). `parsePath` parses `m/44'/.../0/0`; `withIndex` rewrites the final path element for multi-address support (`AddressIndex`/`SignIndex`).
   - **Layer 2 — `sign.go`:** per-curve signers (`signDigest`) and the `Signature` type. secp256k1 = RFC 6979 deterministic, canonical low-S, recoverable; ed25519 signs the message directly; nist256p1 = ECDSA P-256. `Signature` exposes `Bytes()` (64-byte R‖S / ed25519), `Recoverable()` (65-byte R‖S‖V, secp256k1 only), and `DER()`. `Verify(curve, pub, data, sig)` and the symbol-keyed `VerifySignature(symbol, pub, data, sig)` (resolves curve from the registry, mirrors the ECDSA 32-byte-digest contract) verify a signature without any secret.
   - **Layer 3 — `hdwallet.go`:** the public `Sign`/`SignIndex`/`PublicKey`/`Address` methods, all routed through `withCoin` (resolve symbol+index → open seed once → run callback).
   - **Encoders** are split by curve: `encoders_secp256k1.go` (BTC/EVM/Cosmos/XRP/Tron/…), `encoders_ed25519.go` (SOL/XLM/SS58/…), `encoders_nist256p1.go` (NEO). `crypto.go` and `codec.go` hold shared hash/base58/bech32/base32 primitives.

### The hashing asymmetry (don't break this)
ECDSA chains (secp256k1, nist256p1) sign a **32-byte digest** — the *caller* pre-hashes with the chain's hash function. ed25519 signs the **raw message**. `Sign` validates 32-byte input for ECDSA (`ErrInvalidDigest`) and passes ed25519 messages through unchanged. This is inherent to the cryptography, not a style choice.

### Feature layers built on top (Trust Wallet Core parity)
These reuse the derive/sign/encoder layers without touching the secret-isolation core:
- **Curves (`cardano.go`, `curve25519.go`, `ed25519_blake2b.go`, `starkex.go`, `sr25519.go`, `curve_helpers.go`):** the 5 curves beyond the original 3. Dispatched from `derive.go`/`sign.go`. Cardano (ADA) is now **registered** (see below) — its Icarus master comes from BIP-39 *entropy* (not the seed), threaded through a dedicated entropy enclave; the seed-based `withPrivateKey` path still returns `errCardanoNeedsEntropy` as a guard (it must never be fed the seed). StarkNet (STRK) is now **registered** (curve `Starkex`, EIP-2645 path `m/2645'/…`), its sign-key/pubkey vector pinned in `curves_twc_test.go`; only sr25519 remains provisional/unverified and not wired to a registered chain.
- **Cardano (ADA) registration (`cardano.go`, `secret.go`, `hdwallet.go`, `derive.go`, `registry.go`, `address_validate.go`):** ADA (`m/1852'/1815'/0'/0/0`, CIP-1852, curve `Ed25519ExtendedCardano`, 6 decimals) derives its Icarus master from the BIP-39 **entropy**, which `secret` now seals in its own memguard enclave (`secret.entropy`) alongside the seed/mnemonic — populated by `deriveSeedAndEntropyEnclaves` via `bip39.EntropyFromMnemonic`, accessed through `withEntropy(fn)` (returns the exported `ErrNoEntropy` when there is no mnemonic), and wiped in `destroy()` exactly like the seed. A Cardano **base** address (`addr1…`) needs two keys — the payment key (role 0) and the staking key (role 2, `m/1852'/1815'/0'/2/0`) — so `withLeafPublicKey`/`AllAddressesAt`/`AddressRange` route ADA through `withCardanoCombinedPublicKey`/`cardanoCombinedPublicKey`, assembling the 128-byte `ED25519Cardano` pubkey (payment A‖chain ‖ staking A‖chain) and `encodeCardano` (header `0x01` ‖ blake2b224(payment) ‖ blake2b224(staking), bech32 HRP `addr`, no length cap → `DecodeNoLimit` in the validator). `Sign(ADA)` signs with the 96-byte payment extended key. Key-only/WIF/xpub wallets have no entropy, so ADA `Address`/`Sign` fail cleanly. **AllAddresses policy:** ADA **is included** in `AllAddresses`/`AllAddressesAt`/`AddressRange` — each opens the entropy enclave within the existing lock window (`cardanoAddressAt`) rather than the seed, keeping the registry count complete and the `AllAddressesAt`-vs-`AddressIndex` invariant intact. Pinned byte-for-byte to TWC's Cardano `AddressTests.cpp` mnemonic vector (`civil void tool … flash` → `addr1q94zzrtl32tjp8j96auatnhxd2y35fnk6wuxqvqm9364vp9spdkjdsmyfhvfagjzh4uzp9zs6p5djw89jac2g0ujs2eqsuy7pu`) in `cardano_vector_test.go`. Cardano **transaction** signing remains roadmap.
- **Passphrase + custom derivation (`secret.go`, `hdwallet.go`, `path.go`):** `FromMnemonicWithPassphrase`/`FromMnemonicBufferWithPassphrase` thread a BIP-39 passphrase through `deriveSeedEnclave` (passphrase wiped after seed derivation). `AddressPath`/`SignPath`/`PublicKeyPath`/`WithPrivateKeyPath`/`PrivateKeyPath` take an absolute path; `AddressAt`/`SignAt`/`PublicKeyAt` take account/change/index. All route through `withLeafPrivateKeyPath`→`deriveLeafSeedMode` (seed-only; key wiped on return).
- **Mnemonic length + bulk index (`hdwallet.go`):** `NewHDWalletWithWordCount`/`NewHDWalletWithEntropy`/`GenerateMnemonicWithWordCount` generate 12/15/18/21/24-word mnemonics (128–256-bit; `NewHDWallet`/`GenerateMnemonic` stay 12-word wrappers, `ErrInvalidWordCount` on a bad count). `AllAddressesAt(index)` derives every coin at an arbitrary index in one seed-open window (via `withIndex`); `AllAddresses` == `AllAddressesAt(0)`. `AddressRange(symbol, start, count)` derives one coin across a contiguous index range in a single seed-open window for account/gap-limit discovery (seed-only; same guards).
- **Private-key mode (`privatekey.go`, key-only branch in `secret.go`/`hdwallet.go`):** `FromPrivateKeyBytes`/`FromPrivateKeyBuffer` build a key-only wallet; `WithPrivateKey(fn)` (wiped on return) and `PrivateKey()` (memguard buffer) export — mirroring `WithMnemonic`/`Mnemonic`. **The "keys never leave the package as a raw `[]byte`" invariant still holds**; there is no raw getter. `withCoin` was unified into `withLeafPrivateKey` (handles both seed and key-only modes). `validatePrivateKey` accepts every 32-byte-scalar curve (Cardano excluded — 96-byte/entropy).
- **WIF + extended keys (`wif.go`, `extkey.go`):** `FromWIF`/`WithWIF`/`WIF` (secp256k1 Bitcoin WIF); `AccountXPub`/`WithAccountXPrv` export BIP-32 extended keys, and `WatchOnlyFromXPub` → `WatchWallet` derives addresses from an xpub with no seed. Extended keys / watch-only are **secp256k1-only** (`ErrExtKeyUnsupportedCurve`); xprv/WIF export follow the wiped-callback / memguard discipline (no raw-string secret getter).
- **EVM tooling (`eth_rlp.go`, `eth_abi.go`, `eth_eip191.go`, `eth_eip712.go`, `eth_message.go`):** pure-Go RLP, contract ABI, EIP-191 `personal_sign`, EIP-712 typed data, and an `EthereumMessageSigner`-style API. No new deps; reuses `keccak256` + the secp256k1 signer.
- **Non-EVM message signing (`message_bitcoin.go`, `message_solana.go`):** `SignBitcoinMessage`/`VerifyBitcoinMessage` (Bitcoin `signmessage` standard — magic-prefixed `sha256d`, secp256k1 recoverable, base64) and `SignSolanaMessage`/`VerifySolanaMessage` (raw-ed25519 off-chain message, base58). Each pinned byte-for-byte to its Trust Wallet Core `MessageSigner` vector. **Cosmos ADR-36** (`message_cosmos.go`) signs an arbitrary-message amino-JSON `StdSignDoc`/`MsgSignData` with a secp256k1 recoverable signature; the caller-supplied `signer` is validated as bech32 before being embedded (no JSON injection). It has no authoritative external vector, so it is anchored by a self-consistency round-trip + a skipped vector-pin test (`message_cosmos_test.go`). **Tron TIP-191** (`message_tron.go`) signs `"\x19TRON Signed Message:\n32" || keccak256(msg)` (secp256k1 recoverable, pinned to TronWeb), and the chain-neutral `SignRawMessage` (`message_generic.go`) routes ed25519 raw-message vs secp256k1 32-byte-digest signing through the existing `Sign` primitive.
- **Transaction signing (`tx.go` dispatcher + `tx_ethereum.go`/`tx_tron.go`/`tx_ripple.go`/`tx_cosmos.go`/`tx_solana.go`/`tx_bitcoin.go`/`tx_utxo.go`/`tx_zcash.go`, protos in `txproto/`):** `(*HDWallet).SignTransaction(symbol, index, proto.Message)` returns a signed, serialized raw tx (**no broadcast**). Family routing is data-driven from `tx_families.go` (`evmTxChains`/`cosmosTxChains`/`ethermintTxChains`) so all EVM and standard Cosmos chains sign. Ethermint-keyed Cosmos chains sign with an eth_secp256k1 key — keccak256(SignDoc) digest + chain-specific pubkey type URL (`signCosmosEthermintTx`, driven by the `ethermintPubKeyTypeURLs`/`ethermintUncompressedPubKey` maps); **EVMOS** and **INJ** are wired (each pinned byte-for-byte to its TWC vector) — EVMOS uses `/ethermint.crypto.v1.ethsecp256k1.PubKey` with a compressed key, while **INJ** uses `/injective.crypto.v1beta1.ethsecp256k1.PubKey` with an **uncompressed** (65-byte) key (both the type URL and the compressed/uncompressed encoding enter the signed bytes, so each is chain-specific); CANTO/ZETA/ONE stay roadmap pending each chain's own vector. Shapes: EVM legacy/1559/2930 + **EIP-4844** (type-3 blob tx: `max_fee_per_blob_gas` + `blob_versioned_hashes`; network-wrapper blobs/commitments/proofs out of scope) + **EIP-7702** (type-4 set-code tx: `authorization_list` carrying pre-signed delegation tuples `[chain_id, address, nonce, y_parity, r, s]`) + ERC-20 + `ContractGeneric` (arbitrary call) + deploy (empty `to`) + **EIP-2930/1559/4844/7702 access lists** (exported `EthTxModeLegacy`/`EthTxModeEIP2930`/`EthTxModeEIP1559`/`EthTxModeEIP4844`/`EthTxModeEIP7702` for `tx_mode`, `SigningInput.access_list`, vectors pinned to the go-ethereum reference signer — EIP-4844 against `types.NewCancunSigner`, EIP-7702 against `types.NewPragueSigner`, both with go-ethereum v1.15.11); Cosmos `MsgSend`/`MsgDelegate`/`MsgUndelegate`/`MsgWithdrawReward` + multi-message; Solana system + SPL `TransferChecked`; Tron `TransferContract` + TRC-10 `TransferAssetContract` + generic `TriggerSmartContract` (arbitrary call: `call_value`, `data`, `call_token_value`, `token_id`) + legacy Stake 1.0 `FreezeBalanceContract`/`UnfreezeBalanceContract` (on-chain fields 1/2/3/10/15) + `UnfreezeAssetContract` + `WithdrawBalanceContract` + `VoteAssetContract` (repeated vote addresses); plus **raw_json signing mode** (`SigningInput.raw_json` accepts a node/DApp pre-built transaction JSON with `raw_data_hex`+optional `txID`, verifies sha256(raw_data)==txID as a fund-critical guard, and signs — wallet-connect flow). **Bitcoin** tx (`tx_bitcoin.go`) signs BTC/LTC spends across all four standard single-key input types — **legacy P2PKH**, **nested P2SH-P2WPKH** (BIP-49), **native P2WPKH** (BIP-143), and **Taproot key-path** (BIP-341/BIP-340 Schnorr) — outputs to any address type, deterministic in-order coin-selection — pinned against `btcd` (P2PKH/P2SH-P2WPKH/P2WPKH byte-identical; Taproot sighash + BIP-340 verify) and the BIP-143 spec vector. The same Bitcoin engine also signs the non-SegWit UTXO altcoins — **DOGE/DASH** (legacy P2PKH, btcd-oracle-pinned), **BCH** (BIP-143 + `SIGHASH_FORKID`; CashAddr/base58 output decode in `tx_utxo.go`), and **ZEC** (transparent Sapling v4 + ZIP-243 sighash via a self-contained personalized BLAKE2b in `blake2b_personal.go`; `tx_zcash.go`) — routed via `utxoTxChains` and each pinned to a Trust Wallet Core AnySigner vector. The same engine, with **per-coin address params only** (no new signing logic), also signs additional Bitcoin-codebase UTXO altcoins that already derived addresses: native-SegWit **DGB/SYS/VIA/STRAX** (standard BIP-143 P2WPKH; `btcAddrParams`) and legacy-P2PKH **QTUM/RVN/FIRO/MONA/PIVX** (standard pre-segwit double-SHA256; `utxoOutParams`), each proven byte-for-byte by the `btcd`-oracle re-sign in `tx_utxo_altcoins_test.go` (btcd signs a P2PKH/P2WPKH script identically regardless of the coin's address version). Deliberately **deferred** (the standard engine would emit a wrong, fund-losing signature; left out of `utxoTxChains` with a roadmap note): **GRS** (Groestl-512 sighash), **BTG** (BIP-143 `SIGHASH_FORKID` 79), **BCD** ("sbtc"-tagged sighash), **XEC** (BCH-style FORKID path, not wired off `tx_bitcoin.go`), **KMD/ZEN/FLUX** (Zcash-derived ZIP-143/243 BLAKE2b sighash; ZEN also adds a CHECKBLOCKATHEIGHT scriptPubKey suffix), and **NEBL/XVG** (Peercoin-style PoS `nTime` field). The remaining ethermint Cosmos chains (CANTO/ZETA/ONE) remain deferred. Non-EVM wire formats are hand-built (`protowire` for the proto families; raw Bitcoin wire for tx) for byte-exactness.
- **Transaction decoding (`tx_decode_ethereum.go`/`tx_decode_bitcoin.go`/`tx_decode_cosmos.go`/`tx_decode_solana.go`/`tx_decode_tron.go`/`tx_decode_ripple.go`):** `DecodeEthereumTx`/`DecodeBitcoinTx`/`DecodeCosmosTx`/`DecodeSolanaTx`/`DecodeTronTx`/`DecodeRippleTx` parse a signed raw tx back into typed `*…TxFields` (the reverse of the signers) for "what am I signing" display — no keys. Each is anchored by round-tripping the signer's own (TWC-pinned) output; malformed input returns `ErrTxDecode` and never panics.
- **Address validation (`address_validate.go`):** `IsValidAddress`/`ValidateAddress`/`ParseAddress`/`AddressFromPublicKey` via a separate validator registry (reverse of the encoders) — keyed by `Symbol`, **not** stored on the `Coin` struct.
- **Bitcoin address types (`address_types.go`):** `BitcoinAddress(symbol, type, account, change, index)` derives P2PKH/P2SH-P2WPKH/P2WPKH/P2TR (BIP-44/49/84/86) for BTC/LTC without adding registry rows (the registry default stays native SegWit, so `AllAddresses`/the 134 count are unchanged); pinned to the official BIP-44/49/84/86 vectors. The BTC/LTC validators accept all four formats, and `bitcoinDecodeScript` maps any of them to a scriptPubKey for tx outputs.
- **Additional single-chain tx families (`tx_xlm.go`, `tx_algo.go`, `tx_aptos.go`, protos in `txproto/{stellar,algorand,aptos}/`):** dispatched directly from `tx.go` (`familyStellar`/`familyAlgorand`/`familyAptos`). **XLM** Payment uses a `TransactionV0` XDR envelope with the `KEY_TYPE_ED25519` discriminant inserted only into the `SHA256(networkId‖ENVELOPE_TYPE_TX‖xdr)` signing preimage (pinned to TWC `Stellar/TransactionTests.cpp`); **ALGO** Payment signs `"TX" ‖ canonical-msgpack(txn)` directly with ed25519 — no pre-hash (pinned to TWC `Algorand/SignerTests.cpp`); **Aptos** signs `sha3_256("APTOS::RawTransaction") ‖ BCS(rawTxn)` but serialises only `BCS(rawTxn) ‖ authenticator`, with the sender address as `SHA3-256(pubKey‖0x00)` per AIP-55 (pinned to TWC `Aptos/TWAnySignerTests.cpp`). All ed25519 — they sign the raw message. **ADA (Cardano) tx signing remains roadmap** (CBOR body + payment-extended-key signing).
- **Bitcoin multisig (`tx_bitcoin_multisig.go`, `tx_cosmos_multisig.go`):** P2SH and P2WSH **m-of-n** spend layered on the BIP-174 PSBT machinery — `BuildMultisigRedeemScript` (OP_m ‖ BIP-67-sorted pubkeys ‖ OP_n ‖ OP_CHECKMULTISIG), `MultisigP2SHAddress`/`MultisigP2WSHAddress`, and `BuildMultisigPSBT`→`SignMultisigPSBT`→`FinalizeMultisigPSBT`→`ExtractMultisigTx`. Each signer adds only its own partial sig (skips inputs whose script lacks its pubkey); the fund-critical guard verifies `hash160(redeemScript)` matches the spent output's pkScript before signing. Pinned to a `btcd` txscript re-sign (byte-identical DER) + BIP-67. Nested P2SH-P2WSH deferred. **Cosmos LegacyAminoMultisig is a roadmap stub** (no authoritative vector).
- **Bitcoin transaction planning + fee (`tx_bitcoin.go`, `tx_bitcoin_fee.go`):** the `bitcoin.proto` `SigningInput` carries the full coin-selection surface — `byte_fee` (sat/vByte), `input_selector` (`SELECT_IN_ORDER`/`ASCENDING`/`DESCENDING`/`USE_ALL`), `use_max_amount` (send-all), `use_max_utxo`, `disable_dust_filter`/`fixed_dust_threshold` (dust policy), `lock_time`, per-UTXO `out_point_sequence` (opt-in **RBF**, `BTCSequenceRBF` = 0xFFFFFFFD) and `key_index` (sweep multiple derivation indices in one tx), `extra_outputs` (multiple recipients), and `output_op_return` (single ≤80-byte data output). `planBitcoinTx` does deterministic selection/change; `PlanBitcoinTx` previews the plan (selected UTXOs, fee, change, vsize) **without** a seed; `EstimateTxVsize`/`EstimateBitcoinFee` are coarse per-type fee estimators. SIGHASH `ALL/NONE/SINGLE/ANYONECANPAY` (and BCH FORKID) are honoured for the legacy and BIP-143 sighashes; Taproot key-path is `SIGHASH_DEFAULT` only.
- **Bitcoin PSBT (`tx_bitcoin_psbt.go`, `tx_bitcoin_psbt_v2.go`):** BIP-174 (v0) build/sign/finalize/extract is a thin wrapper over `btcd/btcutil/psbt` (`BuildPSBT`/`SignPSBT`/`FinalizePSBT`/`ExtractPSBTTx`) — finalize+extract of a P2WPKH PSBT is byte-identical to `SignTransaction`; supports P2WPKH/P2SH-P2WPKH/P2TR + legacy P2PKH (via a synthetic NonWitnessUtxo). BIP-370 (v2) is hand-serialized (the btcd lib is v0-only) with the same btcd sighash primitives (`BuildPSBTV2`/`SignPSBTV2`/`FinalizePSBTV2`/`ExtractPSBTV2Tx`); v2 is segwit-only (no legacy P2PKH). Both reuse the wiped-on-return key discipline.
- **Bitcoin Taproot script-path + Ordinals/BRC-20 (`tx_bitcoin_taproot_script.go`, `tx_bitcoin_inscription.go`):** BIP-341/342 script-path primitives — `tapLeafHash`, `taprootOutputKey`, `taprootControlBlock`, `tapscriptSighash` (delegates to btcd's `CalcTapscriptSignaturehash`), and `signTaprootScriptPath` (signs with the **untweaked** internal key, the key invariant). On top of them, a two-phase BRC-20 transfer flow: `BuildBRC20Commit` (commit pays a P2TR address committing to the inscription leaf) → `SignBRC20Reveal` (script-path spend revealing `[sig, leafScript, controlBlock]`); `BuildInscriptionScript`/`BuildBRC20TransferBody` build the Ordinals envelope. There is no general "spend an arbitrary Taproot script-path UTXO" entry yet — only the reveal flow is wired.
- **Bitcoin Babylon staking (`tx_bitcoin_babylon.go`):** pure builders for Babylon BTC-staking **outputs** — a 3-leaf (timelock/unbonding/slashing) P2TR tap-tree over the BIP-341 NUMS internal key, plus the `bbn`-tagged OP_RETURN identifier. Output-build side only; spending the staking paths is roadmap, and no authoritative TWC vector exists yet (tests skipped/marked for future pinning).
- **Amount helpers (`amount.go`):** `FormatAmount`/`ParseAmount(symbol, …)` over the registry's native `Decimals`, plus decimals-explicit `FormatUnits`/`ParseUnits` for caller-supplied token decimals, and `NativeDecimals(symbol)`. `big.Int`/decimal-string math only (no float).
- **Provider contract (`providers.go`, `doc.go`):** small interfaces a client implements (`NonceProvider`, `UTXOProvider`, `FeeOracle`, `RecentBlockhashProvider`, `Broadcaster`) plus a per-family `SigningInput`-field→data-source matrix in the package doc. The library never calls these — they are typing/documentation for the "no network I/O" seam.
- **Broadcast payload (`tx_broadcast.go`):** `BroadcastPayload(symbol, out)` returns the exact RPC-ready string per family (EVM `0x`-RLP, BTC/UTXO raw hex, Solana base64, Cosmos base64 `tx_bytes`, Tron TronGrid JSON, XRP `tx_blob` hex) — pure formatting over existing `SigningOutput` fields, no re-sign/no network; mirrors `tx_txid.go` dispatch.
- **Mnemonic UX helpers (`mnemonic_helpers.go`):** pure-function `WordlistPrefix` (autocomplete), `IsValidWord`, `SuggestFinalWords` (all valid checksum completions of an 11/14/17/20/23-word prefix, validated against go-bip39), `MnemonicStrength`. For entry-screen assistance only — never the live secret path.
- `crypto.go` is excluded from gosec via `-exclude-generated` only for the generated `txproto/*.pb.go`; three hand-written safe int→byte conversions carry narrow `#nosec G115` notes. The EIP-4844/7702 vector oracle in `_oracle/` is a separate module (its own `go.mod`; the `_` prefix keeps the Go toolchain from compiling it into the library), so go-ethereum never enters the library's dependency graph.

## How correctness is proven (extend this when adding chains)

Three independent sources of truth — a change to derivation/encoding/signing must keep all green:
- **Encoders** (`encoders_test.go`): every address encoder checked against Trust Wallet Core's `CoinAddressDerivationTests` exact outputs for a fixed key.
- **Derivation** (`slip10_test.go`): ed25519/nist256p1 against official **SLIP-0010** spec vectors; secp256k1/end-to-end against BIP-84 (BTC) and the canonical ETH vector (`hdwallet_test.go`, `address_index_test.go`).
- **Signing** (`sign_test.go`): sign/verify per curve, RFC 6979 determinism, and the strong anchor — sign an ETH digest, `ecrecover`, and assert the address equals the known value for the canonical mnemonic.

The canonical test mnemonic everywhere is the all-`abandon … about` BIP-39 vector (holds no funds). When adding a network, add its Trust Wallet test vector before considering it done.

## CI / release

`.github/workflows/ci.yml` runs Build & Test, Security (govulncheck + gosec), and Lint (golangci-lint v2.12) on push/PR. **Releases are automated**: every push to `main` that passes the gates auto-tags the next semver and publishes to the Go proxy + a GitHub Release. The bump is controlled by the **commit message**: `[major]`, `[minor]`, default = patch, `[skip release]` = none. Pushing to `main` therefore publishes an immutable version — use `[skip release]` if a push should not release. The module path in `go.mod` must always match the repository (`github.com/ranjbar-dev/hd-wallet`), or published versions become uninstallable.

## Conventions

- RIPEMD-160 (`crypto.go`) is intentionally imported despite deprecation (consensus-mandated for Bitcoin/Cosmos `hash160`); it is `//nosec`/`//nolint`-annotated and excluded for SA1019 in `.golangci.yml` for that file only — keep that exclusion narrow.
- Breaking API changes are acceptable pre-1.0 but the security invariants (no exported secrets, keys wiped after use, RFC 6979 + low-S for ECDSA) are not negotiable.
- Commit messages: conventional (`feat:`, `fix:`, …), attribution disabled.
